## Bugs Found (project-specific, not yet in skill/CLAUDE.md)

- **`resolve.dedupe: ['react', 'react-dom']` REQUIRED in `apps/cms/vite.config.ts` when `optimizeDeps.exclude` lists React-dependent packages** — Without dedupe, excluded packages (e.g. `@tanstack/react-router`) resolve their `react` peer dep via Node module resolution while pre-bundled code uses Vite's optimized `react.js`. Result: TWO React instances loaded — `useContext` returns null because Provider was created by Router's React but consumer uses pre-bundled React. Symptom: `TypeError: Cannot read properties of null (reading 'useContext')` in `useRouter`/`useNavigate`/all useContext callers; UI lists render empty (useQuery can't access QueryClient context). NOT classic React duplication (only one version in pnpm-lock) — purely a Vite optimizer dual-resolution issue. Smoking gun in stack trace: same `?v=<hash>` query appears on BOTH `react.js` and `@tanstack/react-router/dist/.../useRouter.js` but resolves through different paths.
- **`@agency/email` barrel re-exports leak `html-to-text` (server-only) into client bundle when consumer imports only pure data constants** — `apps/cms/features/email/types.ts` had runtime `import { AVAILABLE_BLOCKS, DEFAULT_BLOCKS } from '@agency/email'`. ESM resolution pulled the entire `@agency/email` package barrel which re-exports BOTH `./blocks/types` (pure data) AND `./EmailRenderer` (uses `@react-email/components` → `html-to-text`). Vite optimizer can't bundle `html-to-text` (incompatible). Fix: local copy of constants in `features/email/constants.ts` (pure data) + server-only re-export of EmailRenderer in `features/email/render.server.ts`. **General principle**: package barrel re-exports can leak heavy server deps even when consumer only needs pure data — always check what the barrel re-exports beyond the data you import.
- **Plain-dir duplicates in root `node_modules/` silently coexist with pnpm symlinks** — Manual `rm -rf node_modules/{react,react-dom,scheduler,@tanstack/react-query,@tanstack/react-router,@testing-library/react,@testing-library/jest-dom}` was needed before pnpm could rebuild proper symlinks. These plain directories (vs symlinks to `.pnpm/`) were created pre-session by manual package copying during prior debugging. `pnpm install` does NOT detect or fix them — silent coexistence corrupts module resolution. **Affects worktrees too**: vitest watch in worktree crashed with `Cannot find module 'react'` because resolution walked up to MAIN path's plain-dir copies — fix is `rm -rf` on the MAIN repo's plain dirs (not the worktree's). Diagnostic: `ls -la node_modules/<pkg>` should show symlink-to-`.pnpm/`, NOT a real directory. Root `node_modules/<pkg>` should be ABSENT for pnpm-managed packages; only `apps/<name>/node_modules/<pkg>` should exist as symlinks. Confirmed-affected packages: `react`, `react-dom`, `scheduler`, `@tanstack/react-query`, `@tanstack/react-router`, `@testing-library/react`, `@testing-library/jest-dom`.
- **Vite dev server falls back to port 3002 if 3001 occupied — stale dev servers from prior sessions accumulate** — `lsof -i :3001 -i :3002 -P -n | grep LISTEN` before assuming clean state. `pkill -f "vite dev"` sometimes doesn't deliver SIGTERM cleanly; use `kill -9 <PID>` for stuck processes. Browser tabs/bookmarks hold connections to old port → user tests wrong server, sees stale code, false "fix didn't work" reports.
- **`optimizeDeps.exclude` must list ALL TanStack Start router packages, including `@tanstack/router-core`** — Missing `router-core` causes Vite optimizer to lazy-discover it on first browser request, transitively pull `start-server-core/router-manifest.js` through esbuild OUTSIDE the plugin chain. Virtual module `tanstack-start-injected-head-scripts:v` cannot resolve → 500 on lazy split routes (e.g. `$workflowId.tsx?tsr-split=component`). Final required exclude list (5 packages): `@tanstack/start-server-core`, `@tanstack/react-start`, `@tanstack/react-router`, `@tanstack/router-core`. This is upstream-recommended, NOT a workaround. Same root cause re-surfaced after commit `18b8b03` accidentally removed it during shop fix.
- **Vite warm cache (`.vite/deps`) hides broken boundaries after large refactors** — Dev server with `node_modules/.vite` from a prior session returns 200 OK while imports are actually broken. After 21 commits restructuring server/client boundaries, real issues only surfaced after `rm -rf node_modules/.vite apps/cms/node_modules/.vite` + restart. Cold-cache test is the ONLY reliable verification after large refactors.
- **Async Server Component pattern (Next.js RSC) breaks TanStack Start** — `export async function EmailTemplateList()` with `await getEmailTemplates()` in render body crashes. TanStack Start does not support async function components. Always sync component + `useQuery({ queryFn: () => xxxFn() })`. Carry-over bug from Next.js → TanStack Start migration.
- **OAuth callback routes need `server.handlers.GET`, not `loader` + `component`** — Calendar + marketplace OAuth callbacks were using UI route pattern (`loader: async () => throw redirect(...)`, `component: () => null`). Refactor to API route pattern (matches `trigger.ts` gold standard): `createFileRoute(...)({ server: { handlers: { GET: async ({ request }) => Response.redirect(url, 302) } } })`. Idiomatic for third-party callbacks; UI route pattern was a Next.js carry-over.
- **Pure async exports alongside `createServerFn` in same `server.ts` leak server deps to client** — Top-level pure `export async function foo()` in a non-`.server.ts` file pulls server-only deps into client bundle even when the file ALSO has `createServerFn` exports. This is the retry.ts bug, repeated by an agent across 9 handler exports in workflows server.ts. Three valid patterns: (1) inline lambda in `createServerFn().handler(async () => {...})`, (2) handler extracted to `*.server.ts` and imported, (3) NEVER plain async export in non-`.server.ts`. Audit: 22+ files in 15 features had this anti-pattern post-migration.
- **`tenant_roles.name` uses TitleCase ('Admin', 'Member') — and `super_admin` is NOT a role** — `super_admin` lives on `users.is_super_admin BOOLEAN`, not as a row in `tenant_roles`. Migration with `WHERE name IN ('super_admin','admin')` (lowercase) inserted ZERO rows silently — no error, just empty seed. ALWAYS run `SELECT DISTINCT name FROM tenant_roles;` before writing any permission seed migration. Lowercase role names are an AI-default assumption that doesn't match this DB.
- **`workflow_snapshot` JSONB may contain webhook auth tokens — sanitize before returning to UI** — `step_config.headers` on webhook steps can hold Authorization/Bearer tokens. Added `parseWorkflowSnapshot()` helper to strip sensitive headers before sending snapshot to the client. Any new endpoint returning `workflow_snapshot` (or any JSONB containing user-supplied step config) must route through this helper, not pass-through.
- **`Decide Replay Action` guard `prev.id === currentStepExecId` was status-blind — broke retry attempt counter** — In retry mode the Orchestrator reuses the existing latest-attempt row's ID as `currentStepExecId`, so the guard `prev.id === currentStepExecId → continue` fired on EVERY retry of a `failed` step, routing to `continue` instead of `new_attempt`. Symptom: `attempt_number` stayed at 1 after retry; no new audit row created. Fix: narrow guard to `prev.status === 'pending' && prev.id === currentStepExecId` — status is the only correct discriminator between "first execution" (pending row just created) and "retry" (prior attempt is failed/cancelled/running). General rule: any guard that prevents replay-row duplication MUST check `status === 'pending'` together with the ID match — ID alone is ambiguous in retry paths.
- **n8n `onError: continueErrorOutput` requires BOTH the node property AND a wired error connection** — Setting `onError: 'continueErrorOutput'` on an HTTP Request (or any) node is silently inert if the connections JSON does not also wire the error output to a downstream handler (e.g. `Handle HTTP Error` terminal). Symptom: node fails, execution stays in `running` forever in DB because no error path exists in the workflow graph. Fix for Webhook Handler subworkflow: add property + wire error output → terminal node that returns failure status to Orchestrator. ALWAYS verify both ends when adding error handling to n8n nodes; the property alone is a foot-gun.

## Preferences (Universal Stance)

- **`useState` initializer reads `useSearch()` for deep-link auto-open — avoid `useEffect([], ...)` with eslint-disable** — When a deep-linked URL param (`?execution=<id>`) should auto-open a panel on mount, prefer `const [panel, setPanel] = useState(searchParam ? 'history' : 'none')` over `useEffect(() => { if (searchParam) setPanel('history') }, [])`. Initializer reads URL synchronously on mount → no flash, no eslint-disable. Acceptable asymmetry: pasted-URL-AFTER-mount won't sync into state, but URL writes are user-driven from inside the component (via `navigate({ replace: true })`), so the asymmetry never manifests in practice.
- **TanStack Router URL writes for transient panel state MUST use `replace: true`** — Default `push` pollutes history when `?execution=<id>` updates as user clicks rows in a list. `replace: true` keeps back button meaningful (returns to where the user came from, not through every detail click). Pattern: ALWAYS `navigate({ search, replace: true })` for transient/panel/filter URL state; reserve `push` for genuine navigation transitions.
- **Panel-chrome vs view-chrome separation — close button on container, not on each switchable view** — When a side panel hosts switchable inner views (list ↔ detail), put the close affordance on the panel chrome (top-right header), NOT on each view. Keeps close affordance in one consistent location (Linear-style). Saved a duplicate close button in DetailView once recognized.
- **Boy Scout for `useCallback`/`useMemo` is full-file, not partial** — Project rule "no manual memoization, React Compiler does it" means when touching a heavily-memoized file, removing only the NEW additions creates internal inconsistency (mix of memoized + un-memoized). User explicit ("fix all"): full-file sweep is the right call when the file is in active state. Half-fix is worse than either extreme. Safe in this project because dynamic-import boundaries (e.g. `WorkflowCanvas` lazy-imported with `useImperativeHandle` via `canvasRef`) protect heavy renderers from parent handler identity changes — no perf regression risk.
- **"Gramy long term" applies to architecture even in MVP** — User-facing feature can be deferred (MVP cut) but architecture MUST support future extension without rewriting. Cut UI surface, NOT structural decisions that would force a rewrite. Applied: T-209 chose Continuation-with-attempts so adding Fork later is purely additive.
- **"Rób commity co feature" — sequential agent → review → commit per feature for multi-feature refactors** — User mandate after parallel agent dispatch (7 at once) created inconsistent state: 5/7 only added orphan `handlers.server.ts` without updating `server.ts` or removing `queries.ts`. Parallel agents only valid when (a) each scope is fully isolated (no shared files), (b) each agent's work committed before next dispatch, (c) orchestrator verifies claimed result with `git status` (agent reports unreliable — they say "DELETED" while file still on disk). Default to sequential for any refactor touching shared file types across features. Reaffirmed 2026-05-01: 24 atomic commits in single session via sequential dispatch worked cleanly, each independently testable/revertable.

## TanStack Start Server/Client Boundary (Critical)

- **`.server.ts` strip is per-import-statement, NOT transitive** — TanStack Start strips `import { x } from './foo.server'` from client lazy chunks AT THE IMPORT STATEMENT LEVEL. Direct leaf imports (`import { createServerClient } from '@/lib/supabase/server-start.server'`) at top-level in `features/X/server.ts` are fine — bundler removes them from client chunks. BUT: barrel imports with 5+ named imports from `./handlers.server.ts` AND runtime re-exports (`export { dispatchToN8n } from './handlers.server'`) BLOCK the strip — bundler must keep the reference, ESM evaluates the entire graph, AsyncLocalStorage side-effect leaks into client lazy chunk. Symptom: `node:async_hooks` crash on client when lazy split component evaluates.
- **Fix for barrel pattern: dynamic `await import('./handlers.server')` inside createServerFn handler lambda** — Lambda body is stripped from client bundle, dynamic import is "behind" the strip boundary, never evaluated on client. For runtime re-exports use async wrapper functions instead of `export { x } from './handlers.server'`:
  ```ts
  export async function dispatchToN8n(...args) {
    const mod = await import('./handlers.server')
    return mod.dispatchToN8n(...args)
  }
  ```
  Type-only re-exports (`export type { X } from './handlers.server'`) stay top-level — TS erases them.
- **Cross-feature `from '@/features/Y/server'` imports propagate transitively** — `SurveyLinks.tsx` (component in SurveyBuilder lazy chunk) imported from `@/features/workflows/server` — surveys feature crashed transitively through workflows/server.ts barrel chain, NOT through its own server.ts (which was clean). When diagnosing `node:async_hooks` crashes in feature X, always check whether components of X import cross-feature from `feature Y/server` — Y can be the real offender.
- **Decision tree for new feature server.ts imports**:
  - Direct leaf import (`import { x } from 'lib/leaf.server'`) → top-level OK
  - 1-3 named imports from `*.handler.server.ts`, used in 1-3 lambdas → top-level OK
  - 5+ named imports from a `.server.ts` barrel → DYNAMIC import inside lambdas
  - `export { x } from 'foo.server'` (runtime re-export) → ASYNC WRAPPER function
  - `import type { X }` / `export type { X }` → top-level always OK (TS erases)
- **Wrapper pattern is for shared semantics ONLY (decorator), NOT god-switch dispatcher** (correction 2026-05-01) — Wrapper makes sense when 3-5 helpers share cross-cutting concern (auth + audit + cleanup) — encapsulate as decorator like `withAuthAndAudit(permission, fn)`. WRONG to use wrapper as god-switch for 5+ unrelated handlers (`switch (op.kind) { case 'list': ... case 'detail': ... }`) — false abstraction: TS narrowing breaks per-case, single point of failure, hard to debug. For unrelated handlers use **dynamic per-lambda import** (Pattern 3) — self-contained, type-safe, easy add. Ranking: Pattern 1 (direct leaf, 1-4 imports) > Pattern 3 (dynamic per-lambda, 5+ unrelated) > Pattern 2 (decorator, only for shared semantics).
- **Diagnostic flow for `node:async_hooks` crashes**: (1) Read stack trace — which component is top-level. (2) Trace `from '../server'` in lazy split components. (3) Inspect `features/X/server.ts` top-level imports. (4) If barrel from `./handlers.server.ts` (5+ imports) or runtime re-exports → that's the offender. (5) If only direct leaf imports → search cross-feature pull (component of X imports from feature Y server).
- **Path B (single-file refactor) preferred over Path A (prop drilling)** — When fixing async_hooks leak across many components, prefer fixing the source (single `features/X/server.ts`) over prop-drilling values through 8+ components. Path A (prop drilling) is defensive but each new lazy-split component needs new drilling. Path B is single source fix; explicitly chosen 2026-05-01.
- **Falsy starts dropped (lessons)** — (1) Renaming `features/*/server.ts` + `lib/server-fns/*.ts` + `lib/auth.ts` to `*.server.ts` was rejected by Import Protection plugin (UI cannot import from `.server.ts`). createServerFn modules MUST live in plain `.ts`.
