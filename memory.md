## Worktree Gotchas

- **A parallel git worktree can push an uncommitted migration to the SAME staging DB, invisible in your local `migrations/` dir** (2026-07-09, `worktree-venture-scoped-permissions` pushed `20260709120000_venture_scoped_access.sql`). Every later `supabase db push` then errors because staging is ahead of local. Workaround: temporarily COPY the already-applied migration file into the local `migrations/` dir so `db push` sees it as applied and skips it, then remove the copy. Do NOT `migration repair --status reverted` on it — that would falsely un-mark another worktree's genuinely-applied migration.

- **`tsc` in a worktree resolves `@agency/*` to the MAIN checkout, so it shows PHANTOM errors AND hides real defects for UNCOMMITTED package changes** (2026-07-11, `worktree-client-theming` editing `@agency/email`/`@agency/database`). The worktree has no own `node_modules`; a bare `@agency/x` import walks up past the (missing) worktree root to `main/node_modules/@agency/x → main/packages/x` — the MAIN copy, lacking your uncommitted edits. Result: plain `tsc --noEmit` invents errors referencing symbols main lacks, and silently MISSES real type errors in your new package code (a `Required<BlockTypography>` defect passed until forced resolution). vitest/vite resolve correctly (workspace), so tests pass while tsc lies. HOW TO APPLY: to type-check an `@agency/*` change in a worktree, create a TEMP symlink `<worktree>/node_modules/@agency/<pkg>` → `<worktree>/packages/<pkg>` (shadows only that package; others still walk up), run `tsc` from `apps/cms`, then REMOVE it. The true baseline this way was ~31 errors; without both symlinks, phantom counts (55/78) are meaningless. Vercel builds from the branch so it resolves correctly there — the PREVIEW is the real gate.

- **cms VITEST theme-render tests silently run against the MAIN checkout unless `apps/cms/node_modules/@agency/email` is symlinked to the WORKTREE `packages/email` — and the parent `@agency/` dir must exist FIRST** (2026-07-14, vitest+render variant of the tsc trap above). Same walk-up root cause: a bare `@agency/email` import resolves to `main/node_modules → main/packages/email` (branch `main`, lacking unmerged feature work) → theme-token render tests pass/fail misleadingly (two agents mis-reported canvas-theme-token tests purely from this). If the parent dir is missing the `ln` itself fails silently and you never notice. HOW TO APPLY: `mkdir -p apps/cms/node_modules/@agency && ln -sfn "$(pwd)/packages/email" apps/cms/node_modules/@agency/email`, verify `readlink` points at the worktree, run vitest, then `rm -rf` the link.

## Bugs / Architecture Gotchas

- **Debounced autosave MUST flush on unmount, or the last edit before a switch/navigation is silently lost** (2026-07-15, `CampaignTemplateVariablesSection`). The unmount cleanup cleared the debounce timer WITHOUT flushing → a value typed within the 800ms debounce window before a template switch (remounts via `key`) or a route change vanished. FIX pattern: keep the latest pending values + a `saveRef` in refs; the unmount cleanup flushes if a timer is still pending; the debounce callback nulls the timer/pending so the cleanup can't double-save. HOW TO APPLY: any debounced autosave on a component that can remount (keyed) or unmount must flush-on-unmount, not just clear the timer.

- **App-owned email residual-token stripping must cover SUBJECT and body, not just body** (2026-07-15). `stripResidualTokens` ran on the HTML only → an unfilled `{{firstName}}` in the SUBJECT shipped literally. Same leak class as the seeded-subject `{{firstName}}` note below — reinforce, don't duplicate: strip residual `{{tokens}}` on BOTH subject and body for every app-owned send.

- **Server-only code leaks into the CLIENT bundle via a MIXED helper module — a shared module imported by client code must stay PURE** (2026-07-14, `/admin/email-templates` threw "AsyncLocalStorage in client code"). `apps/cms/lib/result-helpers.ts` co-located pure helpers (`parseTemplateVariables`, imported through `email/server.ts` → used by the client `EmailTemplateEditor`) alongside DEAD exports that top-level-imported `@/lib/auth` → `server-start.server` → `@tanstack/start-server-core` → `node:async_hooks`. The `createServerFn` compiler strips only server-fn `.handler()` BODIES; a PLAIN (non-server-fn) export dragged into the client keeps the WHOLE module, and dev Vite does NOT tree-shake → the module's top-level server import EVALUATES in the browser. RULE: any helper module imported (even transitively) by client code must never top-level-import `@/lib/auth` or any `*.server`; put server-only helpers behind a `createServerFn` or in a `.server.ts`. Manifested only via a route using `?tsr-split=component` — a static route-file crawl misses it; found only by a running-dev-server module crawl.

- **New-template default subject seeds a literal `{{firstName}}` that no send context fills** (2026-07-14, found via user testing). `apps/cms/features/email/server.ts` (~line 407) seeds the default subject with `{{firstName}}` → every newly-created email template ships a guaranteed-unresolved `{{firstName}}` token by construction (no code path supplies `firstName`). Fix the seeded default to a token an actual context fills (or none).

- **A content-detected block that early-returns BEFORE a hook makes that hook conditional → React "Rendered more hooks than during the previous render" crash** (2026-07-14, `Canvas.tsx` bonus-list chip). `CanvasBlockRenderer` early-returned a chip when `isBonusListMarkerBlock(block)` (content === `{{bonus_list}}`), then called `useEmailThemeMap()` below. The sample-data toggle rewrites the marker block's content → `isBonusListMarkerBlock` flips false on the SAME fiber → the hook now runs (count changes) → crash. RULE: any hook in a component that content-detects + early-returns MUST be called ABOVE every early return — the detection result must never gate a hook. Applies to any editor that renders a block/node differently based on mutable content.

- **Spread-merge `{ ...selected, ...next }` in an Inspector wrapper SWALLOWS keys DELETED inside a section — a removed key is simply absent from the patch, so the stale value survives the spread** (2026-07-14, `Inspector.tsx`, unremovable theme-color-token / "eternal white border" bug). A TypographySection/BorderSection that does `delete merged[key]` never propagates the deletion up through a spread merge → once a color token was picked it was unremovable. FIX: `applySectionPatch(block, sectionKeys, patch)` — REMOVE all section keys first, then apply only keys present in patch; DELETE (never set `undefined`) because blocks persist to JSONB and code presence-checks fields. Pin the section key list to the `@agency/email` mixin BIDIRECTIONALLY: `satisfies` rejects extra/typo keys but NOT missing ones → add an exhaustiveness assert (`Exclude<keyof Mixin, keys> extends never`) so a future mixin field can't silently resurrect the bug. HOW TO APPLY: any editor section that clears optional keys and merges via spread onto a larger object has this bug class.

## Domain Concepts

- **Token grammar MUST be identical across editor-extraction and send-path substitution, or a token is unfillable in one but stripped in the other** (2026-07-15). The editor's `extractTemplateVariableKeys` used `{{\w+}}` while the send path's `TOKEN_PATTERN` allows surrounding whitespace + dots; a padded `{{ url }}` was invisible to the editor (couldn't offer an input) yet substituted/stripped at send → the value silently vanished. HOW TO APPLY: pin editor-extraction and send-substitution to the SAME token regex (single source), or they drift and a fillable token gets dropped.

- **PRODUCT DECISION (2026-07-15): bonus links are per-campaign email template VARIABLES; the `{{bonus_list}}` auto-splice was removed from EMAILS ONLY** — `so_bonuses` table, the campaign "Bonusy" CRUD, the public landing API (`GET /api/venture/campaigns/:client/:slug`), and the hardcoded fallback builder were KEPT untouched. Bonuses feed TWO surfaces (email + public landing) — this gates any future `so_bonuses` removal: dropping it breaks the landing page even though emails no longer splice it.

- **Variable resolvability for an email template is per-workflow-step `variable_bindings`, NOT per-template-type — and `lib/trigger-schemas.ts` DRIFTS from the real n8n injector** (2026-07-14). The `form_confirmation` catalog lists `clientName`, but `n8n-workflows/scripts/evaluators/get-response-evaluator.js` actually emits `respondentName` (+ `qaContext`/`responseUrl`/…). So for an n8n-SENT template the editor CANNOT prove which `{{token}}` will be filled — the real bindings live in the workflow step, invisible to the editor — and a catalog-driven "unresolvable" warning would misfire BOTH ways (flag a token that IS filled, miss one that isn't). Resolvability is only code-knowable for APP-owned sends (e.g. `venture_bonus` supplies exactly `{companyName}` + the `{{bonus_list}}` structural marker). HOW TO APPLY: a variable-discoverability / unresolved-token feature may warn only for APP-owned send paths; for n8n-sent templates it must be informational-only (cannot assert unresolvable).

## Database / RLS Gotchas

- **`so_clients` uses a COLUMN-SCOPED authenticated SELECT allow-list — every new column is invisible to `authenticated` until the allow-list is re-asserted** (recurring: `sender_name`, `theme`, `theme_id`). Since `20260709163651` the table has `REVOKE SELECT … FROM authenticated` + `GRANT SELECT (explicit column list) … TO authenticated` (to hide plaintext secrets `resend_api_key`/`gmail_app_password`). A bare `ALTER TABLE so_clients ADD COLUMN x` leaves `x` unreadable → CMS `.select()` fails `permission denied for column x` (fails-CLOSED, not open). HOW TO APPLY: any `so_clients` column-add migration MUST append the new column to the `GRANT SELECT (...)` list (copy the prior list verbatim, add the column). `tenants` uses the table-level grant → auto-visible, no re-assert. Secrets stay excluded.

- **Seeding an `email_templates` row: two gotchas, both hit live 2026-07-14.** (a) `ON CONFLICT (tenant_id, type) DO NOTHING` FAILS on staging — `there is no unique or exclusion constraint matching the ON CONFLICT specification` (SQLSTATE 42P10) — despite the CREATE TABLE declaring `UNIQUE(tenant_id, type)`; the arbiter isn't reliably inferable. Use `INSERT ... SELECT ... WHERE NOT EXISTS (SELECT 1 FROM email_templates WHERE tenant_id=… AND type=…)` for idempotent seeds. (b) `email_templates.label` is NOT NULL with NO default (added `20260514120000`) → a seed omitting `label` fails with 23502. HOW TO APPLY: an `email_templates` seed MUST supply `tenant_id` + `type` + `label` (blocks/subject/template_variables have defaults; html_body nullable; theme_id nullable).

- **Per-campaign email template variables live in `so_campaigns.template_variable_values` (flat `Record<string,string>`); this column is behind `so_campaigns`' COLUMN-SCOPED authenticated SELECT allow-list, same as `so_clients`** (2026-07-15). A bare `ADD COLUMN` on `so_campaigns` is invisible to `authenticated` (fails-CLOSED, `permission denied for column`) until appended to the `GRANT SELECT (...)` list — mirror the `so_clients` rule (see above). Resolvability of these values is per-campaign, NOT editor-knowable (see the `variable_bindings` note in Domain Concepts) → editor warnings stay informational-only.

- **`so_campaigns` has NO `tenant_id` column — tenant scope on any `so_campaigns` write is enforced via `assertCampaignOwned`'s campaign→client→tenant walk, NOT a direct `.eq('tenant_id')`** (2026-07-14). HOW TO APPLY: never add or expect a `.eq('tenant_id')` filter on `so_campaigns` queries; gate every campaign write through `assertCampaignOwned` (its ownership walk is the tenant boundary). A field read from an already-owned campaign ROW (e.g. `campaign.email_template_id`) is therefore already tenant-scoped by construction.

- **RLS `INSERT ... RETURNING` fails `42501` when the table's SELECT policy self-references via a SECURITY-DEFINER fn — split into insert + separate read-back** (2026-07-14, verified empirically on prod). `so_clients` SELECT RLS calls a STABLE SECURITY DEFINER `can_access_so_client(id)` that self-queries `so_clients` by the row's id. A Supabase `.insert().select()` (`INSERT ... RETURNING`) then fails `42501 new row violates row-level security policy` for any non-`super_admin`/non-`owner` actor — the definer fn's snapshot can't yet see the just-inserted row. super_admin escapes only because `is_super_admin()` short-circuits WITHOUT reading the row. Hidden for months because only super_admin/owner had ever created clients. HOW TO APPLY: never rely on `INSERT...RETURNING` on a table whose SELECT policy self-references through a SECURITY-DEFINER fn; do `.insert()` (no `.select()`), then read back in a SEPARATE statement (committed snapshot → policy resolves). Deterministic read-back needs a unique key — here `so_clients.slug` (global-unique, migration `20260709180000`) + tenant scope.

- **Every new FK to a delete-guarded reference table MUST be added to that table's reference/usage counter, or delete silently nulls it and the UI lies "0 references"** (2026-07-13). `so_themes` has an app-level delete-guard — `getThemeReferences` + `getThemeUsage` (`apps/cms/features/themes/server.ts`) count usages across `so_clients`/`tenants`/`so_campaigns`. Phase 1 added `email_templates.theme_id` (`ON DELETE SET NULL`) but did NOT extend the guard → deleting a theme used only by an email template would silently null the FK while the UI reported "0 references". Council caught it; fixed by extending both counters + the `ThemeReferences`/`ThemeUsage` types. HOW TO APPLY: when adding any `theme_id` FK (or any FK to a table with an app-level delete-guard), grep `getThemeReferences`/`getThemeUsage` and add the new table (`.eq('tenant_id').eq('theme_id')`).

## Authz Gotchas

- **`ROUTE_PERMISSION_MAP` only gates UI navigation (sidebar + `beforeLoad`); it does NOT protect `createServerFn` endpoints** (2026-07-11, theme CRUD shipped with the `design.themes` route mapping but no server gate → any authed tenant member could HTTP-invoke create/update/delete regardless of role; caught by security validation). HOW TO APPLY: a gated feature needs its OWN server-side check inside every `createServerFn` handler (`requireAuthContextFull()` + `hasPermission(key, permissions)`, mirror the venture `gated()` helper), independent of the route map. Adding the permission key to `ROUTE_PERMISSION_MAP` is not enough.

- **A regex URL-scheme allow-list must strip the chars the BROWSER strips before testing the scheme, or it's bypassable** (2026-07-13). `safeUrlValue` (`packages/email/src/substitute.ts`) first tested a `.trim()`-ed value, but browsers strip ASCII tab `\t`, LF `\n`, CR `\r`, and leading C0 controls `\x00-\x1F` from an href BEFORE resolving its scheme — so `"java\tscript:alert(1)"` / `"\x01javascript:..."` passed the allow-list yet re-collapse to `javascript:` and execute (empirically reproduced). Entity-escaping does NOT save it — those chars aren't escaped and don't break the attribute. FIX: `const cleaned = trimmed.replace(/[\x00-\x20]/g,'')`; test the scheme against `cleaned`, return the original `trimmed` when safe. Regex-based (no `URL` constructor) for n8n-sandbox portability; a `URL()`-based check strips these natively and is safe.

- **`users.role` coarse tier and granular RBAC permission keys are two INDEPENDENT authz axes — venture client create/delete gate on the COARSE column, NOT a permission key** (2026-07-14). `so_clients` INSERT/DELETE (handler `isUnscopedActor` + RLS) require `users.role ∈ (owner,admin)` OR `is_super_admin`, NOT the `venture.clients` key. The `users.role` column is written ONLY by the user editor's "Dostęp do klientów" toggle (`clientAccessToRole`: all→admin, selected→member) — NOT by the role's permission checkboxes. So a custom role with EVERY checkbox ticked but `users.role='member'` still gets "Brak uprawnień" on client create. Self-editing your own tier is blocked by the role-rank guard (SEC-3). HOW TO APPLY: to let someone create venture clients, set their user "Dostęp do klientów" = "Wszyscy klienci" (→ `users.role=admin`); granting permission keys is NOT enough. Scoped members (selected clients) still get full CRUD on ASSIGNED clients — only create/delete of the client ENTITY is unscoped-only.

- **Email `LinkBlock` passes `href` RAW by design — sanitization happens on the FINAL HTML AFTER variable substitution, never per-component** (2026-07-14). Every delivered-mail path sanitizes post-substitution (n8n Send Email Handler `sanitizeHtmlUrls(html)`; venture shared builder `bonus-email-template.ts:241`). WHY late: this also catches `javascript:` smuggled through a `{{variable}}` at send time — sanitizing per-component (before substitution) is insufficient. The CMS live-preview render fn deliberately returns UNSANITIZED html, safe ONLY because `EmailPreview` renders it in an `iframe sandbox=""` (a comment now forbids reuse outside that iframe). Companion to the `safeUrlValue` char-stripping entry above — that entry hardens the sanitizer; this one fixes WHERE it runs.

- **Reusing a permission-gated component on a differently-gated surface needs a client-side mount-guard — don't let its gated query throw** (2026-07-13). `features/themes/ThemePicker` calls `listThemesFn` (hard-gated on `design.themes`, throws before hitting the DB). Reused inside the email editor (gated only `system.email_templates`), a user lacking `design.themes` got the thrown query swallowed to `themes=[]` → misleading empty state ("Brak motywów — + Nowy motyw") linking to a route they can't open. FIX: read `PermissionsContext` client-side; when the component's own gating permission is absent, render a read-only fallback (inherit chip) instead of mounting it. The client check is UX-only — the server gate stays authoritative.

## Grantable CMS Permission Plumbing (making a `system.*` key real)

- **A `PermissionKey` can EXIST in PERMISSION_GROUPS yet be completely INERT — a key is not self-enforcing** (2026-07-11, docforge-licenses). `system.docforge_licenses` existed but only gated sidebar-link visibility; real access was enforced two layers below (RLS + handler `isSuperAdmin` checks) that never READ the key, so granting it did nothing. HOW TO APPLY: before assuming a permission key "works", grep whether any RLS policy / handler guard actually reads it — membership in PERMISSION_GROUPS ≠ enforcement.

- **RLS read-gate and code write-gate must admit the IDENTICAL user set — verify parity explicitly** (2026-07-11). Writes use the service client (RLS bypassed → handler `hasPermission` is the only gate); reads use the RLS-aware client (RLS policy is the only gate). Diverging logic → user can read but not write (or vice-versa). For RLS on GLOBAL tables (no tenant_id, e.g. `docforge_licenses`/`docforge_activations`), mirror `is_super_admin()` with a `current_user_has_permission(text)` SECURITY-DEFINER SQL helper (`LANGUAGE sql STABLE SECURITY DEFINER SET search_path=public`): `is_super_admin() OR users.role IN ('owner','admin') OR EXISTS(user_roles→role_permissions matching key OR its parent group)`. Recursion-safe because DEFINER bypasses RLS on the queried tables and it's never used in a policy on its own source tables. CAVEAT: the SQL helper uses `split_part(key,'.',1)` → matches only a ONE-level parent; app-layer `hasPermission` uses `startsWith` (any ancestor). Identical only for ≤2-segment keys — keep keys shallow or the two gates diverge.

- **`PermissionPicker.tsx` HARDCODES which keys render under each display card — it does NOT auto-render every key from PERMISSION_GROUPS** (2026-07-11). A brand-new `system.*` key is INVISIBLE in the role editor until added to a card's `children` array, AND needs a label in `messages.ts` `permissions[key]` (else falls back to the raw key string). Static validators + build all PASS while the checkbox is simply absent — only caught by opening the role editor in the browser.

- **The SidebarV2 `menuGroups` group is INDEPENDENT of the permission-key namespace** (2026-07-11). A `system.*` key's nav item can sit under the "Management" sidebar group while the role editor lists it under "System" → inconsistent UX. Keep the sidebar group aligned with the key's namespace. Per-item nav visibility is gated by ROUTE_PERMISSION_MAP, NOT the group's `requiredPermissions` (that only gates whole-group visibility).

- **`seedDefaultRoles` grants EVERY ALL_PERMISSION_KEYS entry to each NEW tenant's Admin role — adding a key to PERMISSION_GROUPS auto-flows to new tenants; only EXISTING Admin roles need a backfill migration** (2026-07-11). Consequence under a cross-tenant (Branch A) design: every tenant admin auto-gets the key → cross-tenant visibility. Fine for single-operator Halo Efekt; revisit when a 2nd real tenant onboards.

## Theming System

- **Client/campaign theming design is ratified in `docs/THEMING_DESIGN.md` + `docs/THEME_MANAGER_DESIGN.md`** (2026-07-11). Two LOCKED, expensive-to-reverse rules: (1) **hex is the single source-of-truth encoding** (email needs literal inline hex — mail clients drop CSS vars; web derives `--color-*` at the boundary; never invert); (2) **theme/`so_themes` is NEVER anon-readable** (`so_clients` co-locates plaintext secrets) → public landing gets theme only via a SERVER-SIDE-resolved endpoint, never a direct anon read. Model: named-theme library `so_themes` (whole `tokens JSONB` per row — a themes-AS-rows table, which is compatible with the "no tokens table" rule that forbade tokens-AS-rows) + nullable `theme_id` FK on tenants/so_clients(/campaigns). The resolver `resolveClientTheme({tenantTheme, clientTheme})` stays PURE and unchanged — only the CALLER swaps `read row.theme` → `read so_themes.tokens via theme_id` (helper `lib/theme/fetch.server.ts`, never-throws, fallback to inline `theme`). `so_campaigns.brand` is a LIVE public contract (landing reads it) — do not mutate its shape.

- **The venture bonus send path INTENTIONALLY IGNORES `email_templates.theme_id`** and applies the per-campaign resolved theme (campaign→client→tenant via `resolveClientTheme`) instead (2026-07-14). WHY: per-campaign client branding must win for venture emails; honoring the template's own `theme_id` would override the client's brand. `fetchBonusTemplate` selects only `blocks, subject` — never `theme_id`. So the generic Phase-1 "template bakes its own theme_id" model does NOT apply on the venture path — a deliberate divergence future maintainers must NOT "fix".

- **Venture bonus template resolution is a no-drop chain: `campaign.email_template_id` (read by id, tenant-scoped from the campaign ROW) ?? the `venture_bonus` SINGLETON slug row (tenant default) ?? hardcoded builder** (2026-07-14, Phase 4 model B). `is_default` was ABANDONED — `venture_bonus` stays a singleton unique slug that IS the default. A campaign selects ANY bonus-capable template (blocks contain the `{{bonus_list}}` structural marker); capability is a UI list-filter + warning, NOT a hard DB/type constraint. The shared pure predicate `isUsableTemplateBlocks` (`features/venture/utils/template-blocks.ts`) keeps the read-only effective-send card's resolved template byte-aligned with what ingest actually sends — mirror BOTH the precedence chain AND blocks-usability, or the card lies about what will be sent.

- **The venture send path MUST pass a `ThemeColorMap` to `renderEmailBlocks(blocks, themeMap)` — `applyThemeToCopyBlock` only raw-hexes header/heading/text/footer** (2026-07-14, Phase 4 review). A selectable bonus template can bind a block color to a theme TOKEN via `ThemeTokenSelect` (`textColorToken`/`backgroundColorToken`, e.g. a CTA background=`accent`). The editor preview themes it (it passes a map), but `bonus-email-template.ts` calling `renderEmailBlocks(blocks)` with NO map left token-bound blocks at their default color in the DELIVERED mail. FIX: build the map as `{ ...resolvedTheme }` (same spread the editor's `resolveEmailThemeMap` uses) and pass it; explicit raw hex still wins over the map so the seeded no-token template stays byte-identical. HOW TO APPLY: any programmatic email render off editor-authored blocks must pass the resolved theme map, not rely on per-role raw-hex overlay alone.

- **An email "preview" that doesn't render through the SAME builder as the send WILL diverge from the product — extract a shared builder so preview == send byte-for-byte** (2026-07-14). `ThemePreview` is a generic theme swatch (its header uses the `headerBackground` role), but the real venture bonus email token-binds its header to `primary` via `backgroundColorToken` → the email-template editor themed it (token → orange) while the campaign card's swatch rendered it (role → navy); the ACTUAL send was correct (orange). FIX pattern: shared `mail/render-bonus-email.server.ts` (`resolveVentureSendTheme` + `buildBonusEmailBody`, client passed structurally) used by BOTH ingest (service-role) AND the new owned-path preview fn (cookie/RLS), pinned by a byte-equality parity test. Color ladder `resolveThemedColor`: (a) token ref > (b) raw hex > (c) default token — a block's token binding wins over raw hex. HOW TO APPLY: any "email preview" must render through the same builder as the send; a role-based swatch is NOT an email preview.

## Email / n8n Gotchas

- **n8n `resolve-variables.js` injects user data into `html_body` WITHOUT HTML escaping** (2026-07-13). `n8n-workflows/scripts/evaluators/resolve-variables.js` (~line 39) substitutes `{{token}}` values into the baked `html_body` via `String(value)` with NO escaping. ## Workflow Preferences

- **A "removal" refactor orphans seeded data + dependent classifiers/comments — per-iteration validation misses it; only a full adversarial `/code-review` catches it** (2026-07-15, removing the `{{bonus_list}}` marker-splice from the venture send path). Each iteration's validator passed green, but a subsequent high-effort `/code-review` (24 agents) found the removal (a) orphaned the already-seeded `venture_bonus` `email_templates` row (marker → empty section; needed a data migration), (b) left `resolve-variable-source.ts` / `collectUnresolvableTokens` LYING (bonus_list still badged "auto-spliced"; new per-campaign tokens false-flagged "unresolvable"), and (c) left stale comments. HOW TO APPLY: when REMOVING a mechanism, grep for (i) seeded/migration data that depends on it, (ii) classifiers/badges/warnings that describe it, (iii) comments referencing it — a removal is NOT done when the code compiles. Run a full adversarial review before merge, not just per-iteration validation (which greenlights each step in isolation and misses the cross-cutting orphan).

- **Before designing a NEW per-feature template/CRUD mechanism, grep the feature's `components/` + existing routes for an already-shipped generic CRUD** (2026-07-14, venture Phase 4 — the single most valuable lesson of the session). A 5-member `/council` designed a PARALLEL venture-specific template mechanism (`email_templates.is_default` + `venture_bonus`-as-shared-kind + partial-unique-index relaxation — migration + server layer BUILT & COMMITTED) UNAWARE the email-templates feature ALREADY had full multi-template CRUD via a unique tenant-defined `type`-slug per row (`CreateTemplateDialog` slugifies label→type; `EmailTemplateListClient` lists all as create/delete cards; the `$type` editor opens each) — because NO council member read `features/email/components/`. On discovering it, pivoted to MODEL B (reuse the existing CRUD; campaign selects any bonus-capable template by id) and reverted the model-A over-build. HOW TO APPLY: a council/agent reasoning only from the DB schema + one send path will miss an existing UI CRUD and over-build — always inspect the feature's `components/` and routes for existing CRUD BEFORE proposing new template/CRUD infrastructure.

- **Phased L/XL feature with a ratified design → run `/council` per phase BEFORE implementing that phase** (2026-07-13). For the email-template/variable/theming redesign (`docs/EMAIL_TEMPLATE_ARCHITECTURE.md`) the user directed that EACH phase gets its own `/council` to refine implementation details before any code is written. Generalizes alongside the "validate per iteration" discipline: design deliberation catches cross-cutting issues a single-pass implementation misses (this council surfaced a real security gap + resolved 4 architectural tensions).
