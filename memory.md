- **Cross-site brand tokens via `color-mix(var(--color-X))` in inline style** — Editor (shared CMS package) emits marks with inline `style="background-color: color-mix(in srgb, var(--color-primary) 30%, transparent)"` rather than class names. Each consuming site (website, shop/jacek, shop/oleg) resolves `--color-primary` to its own brand token. WHY: zero coupling editor ↔ consumers, no Tailwind class allowlist needed, preset swatches stay theme-aware across shops.
- **`useEditorState` is the Tiptap v3 way to subscribe** — For reactive `isActive`/`can()`/mark-attribute reads, wrap in `useEditorState({ editor, selector })`. Don't call `editor.isActive()` directly in render — stale value, no re-render on transactions.
- **react-colorful for inline color pickers (not native `<input type="color">`)** — Inline Popover-friendly, drag events don't trigger OS modal, works inside any portal/overlay. Pairs with preset swatch grid for token-mapped quick picks + custom hex/alpha.
- **Workflow retry = Continuation-with-attempts model (2026-04-21)** — Full spec in `docs/guides/workflow/WORKFLOW_RETRY_ARCHITECTURE.md`. Core mechanism: replay guard in `Prepare Current Step` queries latest attempt by status — `completed`/`skipped` reuse cached `output_payload` via `__replay__` virtual step type, `failed`/`pending`/`running` start new attempt with `attempt_number+1`. Per-execution `workflow_executions.workflow_snapshot JSONB` decouples history from live workflow edits. `variableContext` reconstructs from prior `output_payload`s — no staticData persistence needed. WHY chosen over Fork model: Fork (new `parent_execution_id` column) can be added later additively without rewriting Continuation — supports "gramy long term" while keeping MVP minimal.
- **`STEP_OUTPUT_SCHEMA_PRESETS` in step-registry.ts for shared output-shape contracts (2026-04-22)** — When multiple step types need to agree on the shape of an output payload (e.g. AI Action's `qualification_analysis` matching update_response's `save_to_ai_qualification`), define the schema once as a named preset in `step-registry.ts` and reference it from both producer and consumer step configs. WHY: prevents drift between producer output shape and consumer field mapping; preset name becomes the contract identifier in builder UI and runtime validation.
- **Per-action step types: `get_response` + `update_response` (2026-04-22)** — Added to step-registry under the Actions category with panels in `features/workflows/components/panels/`. Follows the per-action rule over polymorphic `save_to_table`. `get_response` hydrates response fields into variable context (replaces implicit trigger hydration); `update_response` persists AI Action output back to the `responses` row. Future Actions (insert_lead, create_contact, ...) follow the same shape.
- **Extract pure computed-value resolvers into `features/<x>/utils/` with tests (2026-04-22)** — `resolveOutputValue` moved from inline component logic to `features/responses/utils/resolveOutputValue.ts` with 16 unit tests. WHY: panel components mixed rendering with path-resolution math; extracting the pure function enables exhaustive edge-case testing (nested paths, missing keys, type coercion) that would be awkward via component tests. Rule-of-thumb: any inline `(a, b) => value` worth three branches of logic gets its own file + vitest.
- **Pre-existing debt: trigger_type duplicated across 4 files (2026-04-22)** — `TRIGGER_TYPES` Set in `panels/index.ts` + raw `trigger_type` string literals hardcoded in 3 other files (step-registry, orchestrator, handler subworkflow). Needs a TRIGGER_REGISTRY single-source-of-truth — follow-up chore task. WHY captured: each time a new trigger type is added (webhook, scheduled, etc.), 4 files must be kept in sync or the workflow silently fails to route; the Set+literals pattern is a drift trap.

## Architecture Decisions (cont.)

- **Parallel registry pattern: keep `StepType` union narrow, add `PLACEHOLDER_REGISTRY` separately (2026-04-28)** — On AAA-T-211 (showcase placeholder steps), rejected adding 19 showcase-only step types to the main `StepType` union in `step-registry.ts`. ## Bugs Found (cont.)

- **Tightening `Record<string, V>` → `Record<UnionType, V>` breaks string-keyed lookup sites (2026-04-28)** — On AAA-T-211, narrowing `PLACEHOLDER_STEP_MAP` from `Record<string, ...>` to `Record<PlaceholderStepType, ...>` for type safety surfaced TS errors at every `MAP[stepType]` call site where `stepType: string` (e.g. `panels/index.ts`, `PlaceholderStepPanel.tsx`). Fix pattern: `(MAP as Record<string, unknown>)[key] !== undefined` for existence checks (membership test doesn't need the value type), and `key as PlaceholderStepType` cast for value access AFTER existence-checking. WHY: TypeScript can't narrow a `string` to a union member just because the index access succeeds at runtime — caller must assert. Don't loosen the map type back to `Record<string, V>`; loosen the access site.

## Preferences (cont.)

- **`disabled` over `readOnly` for non-interactive form fields (2026-04-28)** — Placeholder/showcase fields use the `disabled` attribute, not `readOnly`. WHY: `disabled` removes the field from tab order and signals non-interactivity to assistive tech (screen readers announce "dimmed"); `readOnly` inputs still accept focus, look fully interactive, and read identically to editable fields — users tab into them, try to type, get confused. Rule: if the user has zero way to change the value in this UI state, use `disabled`. Reserve `readOnly` for "value is selectable/copyable but not editable" (rare in this codebase).
- **n8n handler scaffolding: extract evaluator JS to source files, not embed in JSON (2026-04-28)** — `n8n-workflows/scripts/n8n-builder.mjs` `create-handler` command generates 2-node evaluator handler subworkflows by reading evaluator JS from `n8n-workflows/scripts/evaluators/*.js` (source-of-truth) and inlining it into the JSON Code-node body. WHY: handler JSONs are export artifacts — diffing/reviewing logic embedded as a JSON-escaped string is hostile (newlines become `\n`, no syntax highlighting, no linting). Keeping evaluators as standalone `.js` files lets the build step inline them at scaffold time while preserving normal editor tooling. Pattern generalizes: any time you need to ship JS-as-string in a JSON config, write the source as a real file and inline it via build script. Existing handlers retrofitted: condition-evaluator.js, switch-evaluator.js, get-response-evaluator.js.

Edit field is "Nazwa kroku" inside `ConfigPanelWrapper` (parent component above every per-type config panel). WHY here, not as a top-level `workflow_steps.name` column: avoids schema migration for a UI-only concern, every step type inherits renaming for free, and `step_config` is already JSONB so adding optional keys is zero-cost. Pattern: per-step UX metadata (display name, color, icon override) lives in `step_config._*` namespace; runtime-relevant fields stay in dedicated columns.
- **ConditionNode has EXACTLY two source handles (true/false) — custom branch values silently disconnect (2026-04-28)** — Any attempt to use `condition_branch` values other than `'true'`/`'false'` (e.g. `'hot'`/`'warm'`/`'cold'` for lead routing) causes ReactFlow to render the edge as floating/disconnected because `Handle` component IDs in the node only match `'true'` and `'false'`. WHY this matters for switch step design: switch needs N dynamic source handles where each handle's `id` matches `edge.sourceHandle` which maps directly to `workflow_edges.condition_branch` in the DB — the handle ID IS the branch identifier. Don't try to overload condition for multi-branch routing — build switch with dynamic handles instead.

## Architecture Decisions (cont. 3)

- **Trigger Handler is now a pure pass-through; all data hydration via explicit `get_response` (2026-04-29)** — Trigger Handler simplified to 2-node Start → Pass Through. Initial `variableContext` = caller payload + trigger_type only (universal pass-through in Orchestrator's `buildTriggerContext`). The convenience `tenant.companyName` fetch previously embedded in all 4 trigger branches was removed — violated the "no auto-fetch in handlers, all data via variableContext" rule and was redundant with `get_response` already exposing `tenant.name`. Architectural consistency preserved ("every step type has a handler") even though the trigger handler now does no work. WHY: decouples trigger emission from response shape; downstream steps explicitly declare what they need via `get_response` rather than depending on implicit hydration. `get_response` is the canonical source for `companyName` going forward — workflows must NEVER reference `trigger.companyName`.
- **Orchestrator filters `is_active=eq.true` on workflow lookup → inactive workflows fail silently with 500 (2026-04-29)** — `workflows?id=eq.X&tenant_id=eq.Y&is_active=eq.true` means dispatching against an inactive workflow throws "not found or not active" with no specific reason surfaced to the CMS. Mitigation in Test panel: Run Test button now disabled + helper note shown when `is_active=false`. Generalizes: any orchestrator-side filter that excludes an otherwise-valid row should be mirrored in the calling UI as a pre-flight guard, not left as a runtime 500.
- **Synthetic trigger nodes are persisted to `workflow_steps` with `step_type='manual'` etc., not pure visual decoration (2026-04-29)** — Canvas serialization on save includes the trigger node as an actual row. WHY captured: the WorkflowEditor.tsx exception `&& workflow.trigger_type !== 'manual'` skipped synthetic trigger render for manual triggers; combined with AAA-T-211 commit `bfd235a` removing AddNodeDropdown, this left new manual workflows with NO UI affordance to add steps OR triggers (drag-drop also rejects trigger drops by design). Pre-existing regression. Rule: when adding a UI conditional that hides a node-rendering branch, audit ALL entry points for adding steps/triggers — synthetic nodes ARE persistent data, not decoration.
- **n8n Switch node `fallbackOutput: "extra"` adds a 4th unmatched-value output that's invisible from rule count (2026-04-29)** — Reading a Switch node JSON with 3 conditions does NOT mean 3 outputs. `fallbackOutput: "extra"` adds a 4th branch for unmatched values. Easy to miss when reading raw JSON. Affects routing analysis and any "is this exhaustive?" reasoning over Switch nodes (Trigger Handler, Send Email handler, others).

## Bugs Found (cont. 3)

- **RHF default `mode: 'onSubmit'` blocks `reValidateMode` after a manual `trigger()` call (2026-04-29)** — `useForm({ reValidateMode: 'onChange' })` does NOT cause errors to clear on keystroke after `trigger()` runs on mount. `reValidateMode` only activates after `handleSubmit` fires, NOT after a manual `trigger()`. Symptom in workflow config panels: validation banner appears on mount + persists while user types valid input. Fix: explicitly set `mode: 'onChange'` in every workflow panel `useForm` call. Rule: any panel that calls `trigger()` on mount/effect (to prime errors visible at open) MUST also set `mode: 'onChange'` — `reValidateMode` alone is insufficient for the trigger() path.
- **Canvas async vs `selectedNode` sync split causes stale validation in useMemo (2026-04-29)** — `setSelectedNode` is synchronous React state but `canvasRef.getNodes()` reads ReactFlow internal state which updates asynchronously. When a validation `useMemo` re-runs on `selectedNode` change, `canvasRef.getNodes()` still returns the previous node payload → banner shows stale errors for ~one tick. Fix: for the currently-selected node, read `selectedNode.stepConfig` directly (synchronous) instead of going through the canvas ref; only use `canvasRef.getNodes()` for OTHER nodes. Rule: when mixing React state with ReactFlow's internal state in a memo, prefer the React-state source for the focal item to keep the memo's inputs synchronous-coherent.
- **Validation banner in flex-column flow causes layout jump on every keystroke (2026-04-29)** — Conditionally rendering a `<div>banner</div>` inside the canvas flex/column flow shifts everything below by the banner's height — and because validation runs onChange, the canvas jumps by N pixels per keystroke as the banner appears/disappears. Fix: render banner as `position: absolute` overlay inside the canvas's `relative` container, with `pointer-events: none` so it doesn't intercept clicks on underlying nodes. Rule: any UI that toggles visibility on rapid input events (validation, autocomplete, live-preview) must be positioned out-of-flow to avoid visible layout thrash.
- **Orchestrator trusted caller-supplied `tenantId` instead of deriving from workflow row (2026-04-29)** — Trigger route accepted `tenantId` from the request payload and used it directly for the orchestrator dispatch + downstream queries (e.g. `email_templates?tenant_id=eq.${tenantId}` in Send Email Handler). A malicious caller authenticated as tenant A could pass `tenantId=B` and have the workflow execute under B's templates/data. Fix: after auth check + workflow lookup, set `canonicalTenantId = workflowRows[0].tenant_id` and use that downstream for ALL queries. Rule: in any multi-tenant request flow, trust ONLY the tenant ID derived from the authenticated principal's row (or a row owned by them), never the value the caller put in the payload — even if a prior auth check seemed to validate it.

## Architecture Decisions (cont. 4)

- **n8n re-import is mandatory after every workflow JSON edit + credentials must be re-selected (2026-04-29)** — Editing `n8n-workflows/workflows/*.json` does NOT auto-sync to the running n8n instance. Each change requires manual UI re-import + manual re-selection of HTTP/Supabase credentials (n8n drops credential refs on import). Discovered repeatedly during this session's testing. Rule: any time a workflow JSON change is committed, the merge checklist MUST include "re-import in n8n + verify credentials" — there is no shortcut. Long-term mitigation: build an n8n import CLI that preserves credential refs by name (follow-up tooling work).
- **Validation banners use `role="status"` not `role="alert"` for live-updating content (2026-04-29)** — `role="alert"` triggers an immediate AT announcement on every change → screen readers spam users with the same error on each keystroke during onChange validation. `role="status"` (aria-live polite) announces changes when the user pauses, matching the actual UX. Rule: live-updating validation summaries → `role="status"`; one-shot critical errors that block submission → `role="alert"`.

## Bugs Found (cont. 4)

- **TanStack Start `inputValidator` throws synchronously BEFORE handler runs — generic 500 to client (2026-04-29)** — `saveCanvasSchema.step_type: z.enum(CANVAS_STEP_TYPE_ENUM)` rejected placeholder step types (whatsapp_message, send_slack, etc. from showcase templates). Client received generic save error; ALL handler-level logging/validation was DEAD CODE because the request never reached the handler. Symptom: "Nie udało się zapisać workflow" with zero useful diagnostics, even after adding console.logs throughout `validateAllSteps`. Fix: relaxed `step_type` to `z.string().min(1)` in `saveCanvasSchema` so placeholder types pass the wire-format gate; semantic validation still happens in handler. Rule: when debugging a TanStack Start `serverFn` "fix doesn't work" loop, FIRST suspect the `inputValidator` schema — it's a layer ABOVE the handler that fails silently from the handler's perspective. `z.enum(...)` on any field whose values are extensible (step types, trigger types, action types from a registry) is a trap; use `z.string()` at the wire boundary and discriminate inside the handler.
- **Showcase/template workflows have intentionally-null config fields → save schemas must allow `.nullable()` (2026-04-29)** — `sendEmailConfigSchema.template_id` and `aiActionConfigSchema.output_schema` were non-nullable but showcase templates ship with `null` placeholders (user fills them in after import). Without `.nullable()`, the canvas Save button is silently disabled when ANY showcase template node is on the canvas — even if the user hasn't touched it yet. Fix: mark intentionally-empty template fields `.nullable()` in their config schemas. Rule: any step config field that is allowed to be empty in a SHIPPED template MUST be `.nullable()` in its Zod schema, not `.optional()` and not required-with-default — `null` is the explicit "user must fill this" sentinel.

## Process Learnings (cont.)

- **"Add logs immediately, stop guessing" — explicit user override on debugging strategy (2026-04-29)** — Mid-session quote: "wciąż nie działa. dodaj logi bo zgadujesz ciągle."
