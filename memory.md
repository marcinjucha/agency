## Worktree Gotchas

- **A parallel git worktree can push an uncommitted migration to the SAME staging DB, invisible in your local `migrations/` dir** (2026-07-09, `worktree-venture-scoped-permissions` pushed `20260709120000_venture_scoped_access.sql`). Every later `supabase db push` then errors because staging is ahead of local. Workaround: temporarily COPY the already-applied migration file into the local `migrations/` dir so `db push` sees it as applied and skips it, then remove the copy. Do NOT `migration repair --status reverted` on it — that would falsely un-mark another worktree's genuinely-applied migration.

- **`tsc` in a worktree resolves `@agency/*` to the MAIN checkout, so it shows PHANTOM errors AND hides real defects for UNCOMMITTED package changes** (2026-07-11, `worktree-client-theming` editing `@agency/email`/`@agency/database`). The worktree has no own `node_modules`; a bare `@agency/x` import walks up past the (missing) worktree root to `main/node_modules/@agency/x → main/packages/x` — the MAIN copy, lacking your uncommitted edits. Result: plain `tsc --noEmit` invents errors referencing symbols main lacks, and silently MISSES real type errors in your new package code (a `Required<BlockTypography>` defect passed until forced resolution). vitest/vite resolve correctly (workspace), so tests pass while tsc lies. HOW TO APPLY: to type-check an `@agency/*` change in a worktree, create a TEMP symlink `<worktree>/node_modules/@agency/<pkg>` → `<worktree>/packages/<pkg>` (shadows only that package; others still walk up), run `tsc` from `apps/cms`, then REMOVE it. The true baseline this way was ~31 errors; without both symlinks, phantom counts (55/78) are meaningless. Vercel builds from the branch so it resolves correctly there — the PREVIEW is the real gate.

## Database / RLS Gotchas

- **`so_clients` uses a COLUMN-SCOPED authenticated SELECT allow-list — every new column is invisible to `authenticated` until the allow-list is re-asserted** (recurring: `sender_name`, `theme`, `theme_id`). Since `20260709163651` the table has `REVOKE SELECT … FROM authenticated` + `GRANT SELECT (explicit column list) … TO authenticated` (to hide plaintext secrets `resend_api_key`/`gmail_app_password`). A bare `ALTER TABLE so_clients ADD COLUMN x` leaves `x` unreadable → CMS `.select()` fails `permission denied for column x` (fails-CLOSED, not open). HOW TO APPLY: any `so_clients` column-add migration MUST append the new column to the `GRANT SELECT (...)` list (copy the prior list verbatim, add the column). `tenants` uses the table-level grant → auto-visible, no re-assert. Secrets stay excluded.

- **Seeding an `email_templates` row: two gotchas, both hit live 2026-07-14.** (a) `ON CONFLICT (tenant_id, type) DO NOTHING` FAILS on staging — `there is no unique or exclusion constraint matching the ON CONFLICT specification` (SQLSTATE 42P10) — despite the CREATE TABLE declaring `UNIQUE(tenant_id, type)`; the arbiter isn't reliably inferable. Use `INSERT ... SELECT ... WHERE NOT EXISTS (SELECT 1 FROM email_templates WHERE tenant_id=… AND type=…)` for idempotent seeds. (b) `email_templates.label` is NOT NULL with NO default (added `20260514120000`) → a seed omitting `label` fails with 23502. HOW TO APPLY: an `email_templates` seed MUST supply `tenant_id` + `type` + `label` (blocks/subject/template_variables have defaults; html_body nullable; theme_id nullable).

- **Every new FK to a delete-guarded reference table MUST be added to that table's reference/usage counter, or delete silently nulls it and the UI lies "0 references"** (2026-07-13). `so_themes` has an app-level delete-guard — `getThemeReferences` + `getThemeUsage` (`apps/cms/features/themes/server.ts`) count usages across `so_clients`/`tenants`/`so_campaigns`. Phase 1 added `email_templates.theme_id` (`ON DELETE SET NULL`) but did NOT extend the guard → deleting a theme used only by an email template would silently null the FK while the UI reported "0 references". Council caught it; fixed by extending both counters + the `ThemeReferences`/`ThemeUsage` types. HOW TO APPLY: when adding any `theme_id` FK (or any FK to a table with an app-level delete-guard), grep `getThemeReferences`/`getThemeUsage` and add the new table (`.eq('tenant_id').eq('theme_id')`).

## Authz Gotchas

- **`ROUTE_PERMISSION_MAP` only gates UI navigation (sidebar + `beforeLoad`); it does NOT protect `createServerFn` endpoints** (2026-07-11, theme CRUD shipped with the `design.themes` route mapping but no server gate → any authed tenant member could HTTP-invoke create/update/delete regardless of role; caught by security validation). HOW TO APPLY: a gated feature needs its OWN server-side check inside every `createServerFn` handler (`requireAuthContextFull()` + `hasPermission(key, permissions)`, mirror the venture `gated()` helper), independent of the route map. Adding the permission key to `ROUTE_PERMISSION_MAP` is not enough.

- **A regex URL-scheme allow-list must strip the chars the BROWSER strips before testing the scheme, or it's bypassable** (2026-07-13). `safeUrlValue` (`packages/email/src/substitute.ts`) first tested a `.trim()`-ed value, but browsers strip ASCII tab `\t`, LF `\n`, CR `\r`, and leading C0 controls `\x00-\x1F` from an href BEFORE resolving its scheme — so `"java\tscript:alert(1)"` / `"\x01javascript:..."` passed the allow-list yet re-collapse to `javascript:` and execute (empirically reproduced). Entity-escaping does NOT save it — those chars aren't escaped and don't break the attribute. FIX: `const cleaned = trimmed.replace(/[\x00-\x20]/g,'')`; test the scheme against `cleaned`, return the original `trimmed` when safe. Regex-based (no `URL` constructor) for n8n-sandbox portability; a `URL()`-based check strips these natively and is safe.

- **Reusing a permission-gated component on a differently-gated surface needs a client-side mount-guard — don't let its gated query throw** (2026-07-13). `features/themes/ThemePicker` calls `listThemesFn` (hard-gated on `design.themes`, throws before hitting the DB). Reused inside the email editor (gated only `system.email_templates`), a user lacking `design.themes` got the thrown query swallowed to `themes=[]` → misleading empty state ("Brak motywów — + Nowy motyw") linking to a route they can't open. FIX: read `PermissionsContext` client-side; when the component's own gating permission is absent, render a read-only fallback (inherit chip) instead of mounting it. The client check is UX-only — the server gate stays authoritative.

## Theming System

- **Client/campaign theming design is ratified in `docs/THEMING_DESIGN.md` + `docs/THEME_MANAGER_DESIGN.md`** (2026-07-11). Two LOCKED, expensive-to-reverse rules: (1) **hex is the single source-of-truth encoding** (email needs literal inline hex — mail clients drop CSS vars; web derives `--color-*` at the boundary; never invert); (2) **theme/`so_themes` is NEVER anon-readable** (`so_clients` co-locates plaintext secrets) → public landing gets theme only via a SERVER-SIDE-resolved endpoint, never a direct anon read. Model: named-theme library `so_themes` (whole `tokens JSONB` per row — a themes-AS-rows table, which is compatible with the "no tokens table" rule that forbade tokens-AS-rows) + nullable `theme_id` FK on tenants/so_clients(/campaigns). The resolver `resolveClientTheme({tenantTheme, clientTheme})` stays PURE and unchanged — only the CALLER swaps `read row.theme` → `read so_themes.tokens via theme_id` (helper `lib/theme/fetch.server.ts`, never-throws, fallback to inline `theme`). `so_campaigns.brand` is a LIVE public contract (landing reads it) — do not mutate its shape.

- **The venture bonus send path INTENTIONALLY IGNORES `email_templates.theme_id`** and applies the per-campaign resolved theme (campaign→client→tenant via `resolveClientTheme`) instead (2026-07-14). WHY: per-campaign client branding must win for venture emails; honoring the template's own `theme_id` would override the client's brand. `fetchBonusTemplate` selects only `blocks, subject` — never `theme_id`. So the generic Phase-1 "template bakes its own theme_id" model does NOT apply on the venture path — a deliberate divergence future maintainers must NOT "fix".

## Email / n8n Gotchas

- **n8n `resolve-variables.js` injects user data into `html_body` WITHOUT HTML escaping** (2026-07-13). `n8n-workflows/scripts/evaluators/resolve-variables.js` (~line 39) substitutes `{{token}}` values into the baked `html_body` via `String(value)` with NO escaping. Lead/survey-answer data is attacker-supplied and flows into delivered email HTML → injection / broken-markup risk; unresolved keys are also left as literal `{{key}}` in the sent email. Fix designed in `docs/EMAIL_TEMPLATE_ARCHITECTURE.md` Phase 2: a pure `substituteTokens(html, values)` in `@agency/email` with mandatory escape (promote `bonus-email.ts:43 escapeHtml`), back-ported to the shared n8n evaluator. Verified by the council's correctness auditor (16 tool uses).

- **When one substitution fills BOTH email subject and html, escape at the HTML site ONLY** (2026-07-13). Escaping the plaintext subject double-encodes it (`Smith & Co` → inbox shows `Smith &amp; Co`). The n8n Send Email Handler fix split into `substituteBindingsHtml` (escapes, for html) vs raw `substituteBindings` (subject); same axis in TS `substituteTokens` (HTML-only, no `escape?` flag). n8n evaluators (sandboxed, can't import the TS package) keep a byte-parallel `escapeHtml` copy pinned to the package impl by a golden parity test — duplication is correct, drift caught by the test.
