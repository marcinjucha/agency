## Workflow Preferences

- **"nie uzywaj db dockera" — do NOT apply migrations via `docker exec ... psql` against the local Supabase Docker container** (2026-05-15, AAA-T-232). User explicitly rejected the pattern of running `docker exec -i supabase_db_<dir> psql -U postgres -d postgres < <migration>.sql` + manual `INSERT INTO supabase_migrations.schema_migrations` to register it. WHY: the local Docker DB container is keyed off the MAIN checkout directory name (see "Worktree Gotchas" — `supabase_db_legal-mind`), which makes the workflow fragile across worktrees, leaves the local DB in a hand-edited state divergent from migration history, and bypasses the migration system's bookkeeping (`supabase migration list` becomes a lie until `migration repair` is run). HOW TO APPLY INSTEAD (in order of preference): (1) Trust staging/prod as source of truth — `supabase link --project-ref <id>` + `supabase migration list --linked` to verify state, then `supabase db push` to apply remaining migrations through the proper system. (2) For local schema sync, use `supabase db pull` to mirror remote schema, or `supabase db reset` to replay all migrations from scratch through the CLI (not psql). (3) Regenerate types via `supabase gen types typescript --linked` (against remote) rather than `--local` when possible. (4) If a prior session already did inline psql DDL and `db push` now fails with "column does not exist", use `supabase migration repair --status applied <version>` to register the ad-hoc migration WITHOUT re-running its SQL — never reach for `docker exec psql` as the fix. Generalize: never bypass the Supabase migration CLI by going straight to the Docker container, regardless of how convenient it seems.

- **`git add <explicit-path>` + `git commit` still picks up files already staged in INDEX from prior sessions.** This session's commit accidentally swept in 4 unrelated `docs/sessions/*.md` deletions that were sitting in the index from a previous session. Rule: before staging new work, run `git status` to see what's ALREADY in the index, and `git restore --staged <path>` to drop pre-staged entries that don't belong to the current commit. Specifying paths in `git add` only ADDS — it does not implicitly limit the commit to those paths. The commit's scope is the entire index, not the args to `git add`.

## Lockfile & Vercel CI Deploy Verification

- **Any `package.json` dependency change MUST be followed by a lockfile update before push, or Vercel CI install fails** (2026-06-10). Removing a dep from `apps/website/package.json` WITHOUT updating `pnpm-lock.yaml` PASSES the local build (even `npm run build -- --force`) because the main checkout's `node_modules` still has the package on disk — but FAILS on Vercel with `ERR_PNPM_OUTDATED_LOCKFILE` (CI uses `--frozen-lockfile` by default). Fix: `pnpm install --lockfile-only` at repo ROOT to reconcile the lockfile, then amend + force-push the feature branch. Note the lockfile diff can be surgically small (here 3 lines) because the dep stayed referenced by `apps/cms` — only the `apps/website` importer edge was dropped. WHY: local node_modules is a stale superset of what the lockfile declares; CI installs strictly from the lockfile, so a green local build is NOT proof the install reproduces.

- **Verify the Vercel PREVIEW deploy (status Ready + serves real data) BEFORE merging to prod** (2026-06-10). A local `--force` build going green is NOT sufficient proof for a dependency change or data-layer change — the preview's clean `pnpm install` is the real gate. This session's red preview (`ERR_PNPM_OUTDATED_LOCKFILE`, above) was caught at the preview stage instead of becoming a prod outage. Concrete instance reinforcing the staging-first rule: the preview's fresh install reproduces CI conditions that the main checkout's warm node_modules hides.

- **memory.md / docs MUST be committed onto the feature branch and included IN the merge to prod — not left as a dangling dirty working-tree file** (2026-06-10). This session left `M memory.md` out of the merge; user called it out ("memory.md tez powinno zostac dodane do merga"). Concrete miss reinforcing the existing "All docs commits before merge" rule (root CLAUDE.md): a docs/memory edit that is still dirty in the working tree at merge time does NOT reach main. Before merging, `git status` must be clean of intended-doc files — they belong in a commit on the branch.

## Vercel Edge Caching — TanStack Start (Nitro vercel preset)

- **Edge caching DOES work for TanStack Start on Vercel — the route's `s-maxage`/`stale-while-revalidate` are STRIPPED from the client-facing `cache-control` but HONORED internally by the edge** (2026-06-10). Prod `haloefekt.pl` returns `x-vercel-cache: HIT` with a client-facing `cache-control: public, max-age=60`, while the route sets `s-maxage=3600`. Proof the s-maxage is real: `age: 894` while still `HIT` — age (894s) far exceeds the client `max-age=60`, so the SHARED cache is holding the response beyond the client TTL ⇒ s-maxage is in effect. `x-vercel-id: fra1::iad1` reveals edge=Frankfurt, origin=US. HOW TO APPLY: don't conclude edge caching is broken just because the response `cache-control` shows only `max-age=60` — check `x-vercel-cache` + `age` headers; `age > max-age` while `HIT` confirms s-maxage works.

## Render-Blocking Font CSS — Biggest First-Paint Win

- **Narrowing font `@import` to only rendered weights+subsets beats font preload for first paint** (2026-06-10, haloefekt.pl perf). The biggest concrete win was NOT a `<link rel=preload>` but shrinking render-blocking font CSS: replacing `@import '@fontsource/jetbrains-mono'` (all weights × all subsets) with only the rendered weights+subsets cut built CSS 21.8→14.5 KiB and render-block 660→300ms. Also: `@fontsource-variable/lora` ships NO subset-split CSS entry (only `index.css` / `wght.css` / `wght-italic.css`) — import `wght.css` to drop the italic axis; non-latin subsets stay emitted in the file but are never fetched because `unicode-range` gates the actual download. HOW TO APPLY: for fontsource fonts, import the narrowest weight+subset CSS entry that covers what renders; the `@import 'package'` barrel pulls every weight × subset into the render-blocking critical path.

- **Lighthouse "Reduce unused JavaScript NN KiB" can be a red herring when TBT=0** (2026-06-10). The council correctly flagged that the 84 KiB unused-JS banner does NOT directly affect the Performance score when Total Blocking Time is 0 — JS is not the first-paint bottleneck in that case. Don't chase the unused-JS number when TBT is already 0; the render-blocking CSS / LCP path is where the score moves.

## Shared-Generated HTML Content — Fix at the Generation Source

- **`html_body` is generated ONCE in the CMS and consumed by MANY surfaces — fix shared content at the GENERATION source (Tiptap extension config), never per-consumer at render** (2026-06-10, STRONGEST — user corrected this twice). The editor's `generateHTML` (`apps/cms/features/editor/utils.ts` + extensions in `editor/extensions.ts`) produces `html_body` ONCE; it is then rendered via `dangerouslySetInnerHTML` by blog (`BlogArticlePage`), legal pages (website + both shops), and shop products. A render-time per-consumer fix (transforming the HTML string in one component) SILENTLY MISSES every other consumer. Correct fix = configure the Tiptap extension at the source so generateHTML bakes the attribute into every `html_body`: `Image.configure({ HTMLAttributes: { loading: 'lazy', decoding: 'async' } })`, `Link.configure({ HTMLAttributes: { rel: 'noopener noreferrer', target: '_blank' } })`. CAVEAT: source-config only affects NEWLY-generated content — existing rows keep the old HTML and need a backfill OR a manual re-save (user chose manual re-save for the small blog). HOW TO APPLY: any change to how editor HTML renders (lazy-load, rel attrs, classes, decoding) goes into the Tiptap extension config, then plan for existing-row backfill separately.

## CMS Tiptap Editor & Media Gotchas

- **Base `@tiptap/extension-image` models only `src`/`alt`/`title` and SILENTLY DROPS unknown attrs (width/height)** (2026-06-10). To carry `width`/`height` through to the rendered `<img>` you need `Image.extend({ addAttributes() { ... } })` declaring them — otherwise they vanish from `html_body` and you lose the free CLS fix. `media_items` already stores `width`/`height` (enables intrinsic sizing) but had NO alt column → added `alt_text`.

- **StarterKit v3 BUNDLES `link` + `underline` — adding them explicitly triggers a "Duplicate extension names" warning** (2026-06-10). Disable the bundled versions so the explicitly-configured instances win: `StarterKit.configure({ link: false, underline: false })`, then add your own `Link.configure(...)`. Otherwise the bundled defaults silently override your config.

- **Reusable alt text belongs on the media item (Library), with insert-time fallback `alt_text || name` (never empty)** (2026-06-10). Thread the fallback through ALL insert paths — including `createMediaProxyEditor`. A PATCH builder for partial media updates MUST distinguish key-absent (don't touch) vs explicit null (clear), so a rename-only update doesn't blank the alt. HOW TO APPLY: partial-update builders need three states (absent / null / value), not two.

- **`uploadImageToS3` uploads the raw full-res file 1:1 to S3 — no resize, no webp** (2026-06-10). @unpic is present but NOT actually transforming; it serves the raw PNG with only width/height attrs. Byte-level image optimization (resize + webp on upload) deferred to AAA-T-302. Don't assume @unpic's presence means images are optimized.

## INP / Performance Diagnosis via Chrome Automation

- **INP (Interaction to Next Paint) on navigation that is ~100% presentation-delay = synchronous image decode of unoptimized raw images, NOT hydration/handler/data-layer** (2026-06-10, prod diagnosis). Phase breakdown was inputDelay≈0, processing≈0, presentation≈everything → rules out hydration, React handler render, and the data layer. Cause: article-body `<img>` were raw Tiptap-content images with no `decoding="async"`/`loading="lazy"`, and uploads ship full-res PNGs. Fix layer 1 (timing): `decoding=async` at the Tiptap source (see shared-HTML section). Fix layer 2 (bytes, deferred → AAA-T-302): resize+webp on upload. HOW TO APPLY: the INP phase breakdown (input/processing/presentation) is the key diagnostic — it localizes WHICH lens owns the cost; absence of a matching longtask + all-presentation = paint delayed by layout/decode, not blocking JS.

- **Measuring INP via Chrome automation** (2026-06-10): PerformanceObserver on `event` (Event Timing, filter `interactionId>0`, compute inputDelay/processing/presentation phases) + `long-animation-frame` (LoAF script attribution + forcedStyleAndLayoutDuration) + `longtask`. Trigger REAL clicks via the `computer` tool — CDP input is trusted input → gets an `interactionId`; a JS `.click()` would NOT. CRITICAL: check `document.visibilityState === 'visible'` before trusting magnitudes — a backgrounded automation tab throttles paint and inflates presentation-delay (a 3128ms reading was tab-context-inflated; the qualitative phase-breakdown + which-navigation conclusions stayed valid). Companion to the existing "backgrounded tab throttles" note.

## TanStack Router Staleness — Two Knobs, Not One

- **`defaultPreloadStaleTime: 0` is a latent route-jump bug that React Query was masking; removing RQ exposed it** (2026-06-10). TanStack Router has TWO independent staleness knobs: (1) `defaultPreloadStaleTime` — how long intent-preloaded (hover) data stays fresh; MUST exceed the hover→click gap (used `1000*30` = ~50× margin). (2) route-level `staleTime` — freshness across repeat navigations. With `0`, intent-preloaded data was already stale on arrival → every navigation re-ran the blocking loader and flashed the route's `pendingComponent` skeleton → visible "jump". RQ's own cache had been absorbing the refetch; removing RQ surfaced the flash. ALSO (a /council disproved a hypothesis): removing React Query did NOT cause the INP problem — the data layer runs on navigation transitions, never on in-page pointer events; "combining router+RQ" (`routerWithQueryClient`) would not move INP and only adds client JS.

## Schema Migration & Deploy Ordering

- **Deploy schema-tolerant code BEFORE applying a column-dropping migration to prod — reverse order = prod outage** (2026-06-09). When a migration DROPs columns the currently-deployed prod code still SELECTs, the old code 500s the moment the column vanishes. Correct order: (1) write new code that tolerates BOTH schemas (`select('*')` + optional-chaining/fallback on the dropped field), (2) merge + push, (3) confirm Vercel shows "Ready/Production" on the merge SHA, (4) THEN apply the migration to prod. WHY: prod runs the deployed bundle, not your branch — dropping the column first leaves live code reading a column that no longer exists until the deploy finishes. Schema-tolerant code is the safety margin that lets it run on old AND new schema during the rollout window.

- **`supabase gen types --linked` sweeps in unrelated staging schema drift — `git diff` and surgically keep only the target table** (2026-06-09). Regenerating types after one migration pulled in `media_items.is_downloadable`, iceberg/storage internals, and `__InternalSupabase` metadata far beyond the intended table — all the accumulated un-migrated drift on the linked remote. Rule: after `gen types`, `git diff` the types file, hand-edit to keep ONLY the change for the table you migrated, and `git checkout` the rest. WHY: committing the broad sweep silently bakes phantom/unreviewed schema into the typed surface, and types then claim columns exist that no migration created.

## Visual Rendering Bugs — Static Validators Don't Catch Them

- **Registry/map abstractions must be VERIFIED VISUALLY — static validators (functional + architecture + security) all pass while UI still renders the wrong icon/color/label** (2026-05-21, AAA-T-284 trigger-registry refactor). Iter 3 introduced `TRIGGER_ICON_MAP` in `node-registry.ts` and migrated `TriggerNode.tsx` (claimed to consume it), but the actual JSX retained a hardcoded `<Zap className="..." />` on line 42 — `lookupNodeConfig()` and `TRIGGER_ICON_MAP` were never read at render time. All 3 static validation passes (Phase 3 functional, 3b architecture, 3c security) returned PASS because: (a) imports existed and looked correct, (b) the map was defined and exported, (c) no type errors. The bug only surfaced when a human (or browser automation) looked at the actual rendered icon on the canvas and noticed `form_submitted` triggers showed ⚡ instead of ✋. **HOW TO APPLY:** Whenever an iteration's stated goal is "consume registry map X for UI rendering Y" (icons, colors, labels, badges), Phase 4 manual testing is NON-OPTIONAL and must include a visual diff (browser screenshot, or zoom on the specific element) — code review and validators cannot detect a hardcoded fallback sitting next to a correctly-imported lookup function. Companion rule to "validation-per-iteration discipline": validators verify structure, visual inspection verifies render.

- **Chrome browser automation sequence for visual UI debugging — use `computer zoom` for icon/small-element verification, not `screenshot`** (2026-05-21, AAA-T-284 Phase 4). When debugging trigger node icons on the workflow canvas, the working sequence is: (1) `mcp__claude-in-chrome__tabs_context_mcp` — check existing tabs/group, (2) `tabs_create_mcp` — open new tab in the project's chrome group, (3) `navigate` — to local dev URL (e.g. `http://localhost:5173/...`), (4) `read_page` with `filter: interactive` — get login form / nav structure without full DOM noise, (5) `find` — natural-language element search (`"trigger node icon"`) returns coordinates, (6) `computer` with `action: zoom` and `region: [x0,y0,x1,y1]` — KEY TOOL: zoom into a specific pixel region to clearly distinguish icon shapes (⚡ vs ✋ vs 📅), (7) `computer left_click` + `screenshot` for full-page state when interaction is needed. **WHY `zoom` over `screenshot`:** full-page screenshots compress icons to ~16px squares where ⚡/✋/📅 are visually indistinguishable; `zoom region` blows up that 16px region to a clear, comparable image. **WHY this sequence over a single screenshot:** `read_page interactive` + `find` lets the orchestrator address elements semantically without measuring pixels by hand; the zoom step is what made the TriggerNode hardcoded-icon bug visible in a single round-trip. Captured here because no skill currently documents the visual-debugging branch of Phase 4 manual testing.

- **Traced/exported SVG logo bakes the white background in as a filled `<path>` — it is NOT transparent** (2026-06-09). The exported SVG contained a full-canvas rectangle path with `fill="#FEFEFD"`, so the logo showed an opaque white/cream square on tinted sections. Fix: set that background path to `fill="none"`. Same root cause as a white-bg PNG showing a white box on a colored section — exporters/tracers add an explicit bg fill rather than leaving it transparent. HOW TO APPLY: when a logo shows an unexpected solid box, grep the SVG for a large rect/path with a near-white `fill` and null it out.

- **Backgrounded Chrome automation tab throttles JS timers + IntersectionObserver + CSS animations (`document.hidden===true`) → blank screenshots and false "stuck at opacity-0" reads** (2026-06-09). An unfocused automation tab pauses `setTimeout`, IntersectionObserver callbacks, and animations, so reveal-on-scroll sections below the fold never fire and screenshots come back blank/empty. HOW TO APPLY: when the tab may be unfocused, verify render via `getComputedStyle` on NON-animated DOM nodes (read the actual computed property), not via screenshots — screenshots of a throttled tab lie. Companion to the existing `computer zoom` visual-debugging note.

- **Reveal-on-scroll (ScrollReveal-style) needs three safety nets or sections stay invisible forever** (2026-06-09). Without them, a section reached by a navbar anchor jump (`#wspolpraca`) or a fast scroll can stay at `opacity-0` permanently because the IntersectionObserver entry never crosses the threshold during the jump. Required nets: (1) immediate reveal if the element is already in-viewport on mount, (2) a timeout fallback that force-reveals after N ms, (3) honor `prefers-reduced-motion` (reveal instantly). HOW TO APPLY: any new scroll-reveal animation must ship all three — animation-gated visibility with no fallback is a content-disappears bug, not a polish issue.

## Delivery Method Decision Framework (Post-Pivot)

- **Four-axis delivery framework for every post-pivot feature: CMS native / standalone n8n / mix / Notion delivery** (2026-05-22, Strategic Pivot follow-up). Before designing ANY new feature for Halo Efekt or a client, classify it across four delivery modes — they are NOT interchangeable. (1) **CMS native** — feature lives in `apps/cms/features/...`, owned by us, requires multi-user auth, complex CRUD, custom UI. Wins for: active operational tooling, shop management, content editing. (2) **Standalone n8n** — workflow on a per-client n8n VPS instance, no UI surface beyond n8n's own. Wins for: scheduled jobs, integrations, AI ops, anything async > 2s. (3) **Mix** — CMS triggers n8n via webhook, n8n executes async work, optionally writes back to Supabase or Notion. Wins for: CMS-initiated workflows with heavy backend processing. (4) **Notion delivery** — Notion database is the UI/DB, n8n is the brain, no custom frontend at all. **Wins for: read-mostly dashboards, status toggles, simple CRUD where Notion's grid/board views are sufficient.** **Active multi-user CRUD with auth/permissions wins for CMS.** **How to apply:** when scoping a new feature, walk through the four axes in order and pick the FIRST that fits — defaulting to "CMS native" is the pre-pivot habit and almost always wrong now. Notion delivery is dramatically under-considered: many "client dashboard" / "lead tracker" / "content pipeline" features fit Notion+n8n perfectly with zero CMS code.

## TanStack Start Import-Protection — `.server.ts` Suffix & Production Build Gate

- **`.server.ts` suffix is a TanStack Start import-protection boundary — `createServerFn` WRAPPER files must NOT use it** (2026-07-08, venture bonus-funnel). A createServerFn wrapper named `admin.server.ts` (dotted suffix) is treated as SERVER-ONLY → every client import (routes + components) is DENIED at production build with `[import-protection] Import denied in client environment`. - **dev-server + unit-tests + static validators GREEN ≠ production build green — run `turbo run build --filter=<app>` before declaring a feature done / before merge** (2026-07-08). `npm run dev` does NOT enforce TanStack import-protection, and the functional/architecture/security validators all PASS — so a red production build (the `.server.ts` issue above) shipped unnoticed through iter-5 and was only caught by an actual `turbo build`. This is the OTHER direction of the existing "`vite build` exit 0 ≠ types valid" note (that one adds `tsc --noEmit`; this one adds `turbo build`). ## Webhook Ingest — Secret Placement, Secret Hiding, Idempotency (Tally)

- **Per-campaign signing secret in DB REQUIRES the slug in the URL path, not in the signed body (verify-before-parse chicken-and-egg)** (2026-07-08, venture leads ingest). `/leads/:slug` lets you look up the campaign + its secret BEFORE reading/trusting the body. You CANNOT pair a per-row secret with a slug hidden inside the signed body — you'd need the secret to verify the body, but the secret lives on the row identified by data inside that body. A global env secret is the only variant that pairs with a body-embedded slug.

- **Hide a secret DB column while keeping a UI "is it set?" signal: REVOKE SELECT + re-GRANT an allow-list + a STORED generated boolean** (2026-07-08). `REVOKE SELECT ON <table> FROM authenticated` then `GRANT SELECT (<cols excluding the secret>)`, plus `<flag> boolean GENERATED ALWAYS AS (<secret> IS NOT NULL) STORED` (STORED → readable without SELECT on the base secret column). Secret becomes service-role-read-only; operator still writes it (INSERT/UPDATE untouched); UI reads only the boolean. CRITICAL: the GRANT allow-list must stay byte-identical to the handler's explicit `.select(...)` column list or reads fail-loud with "permission denied for column". Companion to shop_marketplace encryption / app_config patterns.

- **No-lead-drop webhook idempotency: discriminate Postgres `23505` from other insert errors — don't blanket-return 200** (2026-07-08). A handler that catches ALL insert errors and returns 200 conflates a benign `23505` unique_violation (idempotent duplicate — correct to 200, skip side-effects, the concurrent winner handles them) with a genuine transient DB failure (lead NOT written — must return non-2xx so the provider retries). Blanket 200 silently DROPS the lead. Inspect `error.code`: 23505 → 200, any other → non-2xx.

- **Tally webhook contract** (2026-07-08): header `Tally-Signature` = HMAC-SHA256 base64 over the RAW request body — verify raw text BEFORE `JSON.parse`. Payload `{ eventType:"FORM_RESPONSE", data:{ submissionId, fields:[{key,label,type,value}] } }`. A CHECKBOXES question emits BOTH an aggregate row (value = array of selected option ids) AND per-option boolean rows (`<key>_<optId>`) — a consent extractor must handle both. Tally submission ids are unique only WITHIN a form → multi-form ingest needs a composite dedup key `(campaign_id, submission_id)`, not a global unique.
